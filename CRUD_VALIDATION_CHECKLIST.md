# CRUD Validation Checklist

هذه القائمة هي معيار الفحص لأي صفحة أو مودال فيه إنشاء، تعديل، حفظ، ترحيل، قفل، أو إرسال بيانات إلى الـ backend.

- يجب أن يكون لكل فورم validation قبل الإرسال يمنع البيانات الناقصة أو غير الصحيحة من الوصول إلى الـ backend.
- يجب أن تظهر أخطاء الحقول تحت نفس الحقل المعني مثل الاسم، الكود، البريد، التاريخ، الكمية، السعر، الحساب، الدور، أو الولاية.
- يجب أن يظهر خطأ عام واضح داخل نفس الصفحة أو داخل نفس المودال عند رجوع خطأ من الـ backend.
- يجب قراءة أخطاء الـ backend من شكل الاستجابة القياسي `errors[]` وربطها بأسماء الحقول بدل تركها في `console.error`.
- يجب ألا يغلق المودال بعد الضغط على حفظ إلا بعد نجاح عملية الحفظ فعلياً.
- يجب ألا ينتقل المستخدم إلى صفحة أخرى بعد الحفظ إلا بعد نجاح رد الـ API.
- يجب تعطيل زر الحفظ أثناء الطلب وإظهار حالة مثل `جارٍ الحفظ...` لمنع تكرار الإرسال.
- يجب مسح خطأ الحقل عند تعديل قيمته حتى لا يبقى الخطأ القديم بعد التصحيح.
- يجب التحقق من الأرقام قبل الإرسال، مثل الكمية والسعر والرصيد والطاقة الاستيعابية، ومنع القيم السالبة أو غير الرقمية عند الحاجة.
- يجب التحقق من البريد الإلكتروني عند وجوده، وإظهار رسالة واضحة إذا كان غير صالح.
- يجب التحقق من التواريخ، ومنع تاريخ نهاية يسبق تاريخ بداية.
- يجب التحقق من الصفوف المتكررة داخل الجداول مثل أسطر الطلبية أو القيد أو الفاتورة، وليس فقط رأس الفورم.
- يجب تحويل أسماء الحقول في payload إلى الأسماء التي ينتظرها الـ backend مثل `sku`, `baseUnitId`, `categoryId`, `roleId`, `warehouseType`.
- يجب استخدام helper مشترك لمعالجة أخطاء الـ API حتى يكون السلوك موحداً بين الصفحات.
- يجب بقاء الصفحات السليمة كما هي وعدم تعديلها إذا كانت تحقق كل البنود.
- يجب تحسين الصفحة فقط عندما نجد نقصاً فعلياً في الفاليديشن أو عرض الخطأ أو انتظار نجاح الـ API.
- يجب تشغيل build بعد التعديلات للتأكد أن الواجهة لا تحتوي كسر JSX أو import أو typing.
- يجب اعتبار رسائل التحميل فقط مثل جلب القوائم والتقارير خارج نطاق CRUD إلا إذا كانت تمنع المستخدم من معرفة سبب فشل حفظ بيانات.
