# ALLAL-ARTICLE — Spring Boot Backend Architecture Study

> قرار معماري مبدئي لمرحلة الباك اند المستقبلية.
> الهدف من هذا الملف أن يكون مرجعًا مباشرًا لـ Claude Code عند استلام ملفات الباك اند لاحقًا.

## القرار النهائي المقترح

- الباك اند: `Spring Boot 3.x`
- اللغة: `Java 21`
- قاعدة البيانات: `PostgreSQL`
- إدارة الميغريشن: `Flyway`
- الأمان: `Spring Security`
- أسلوب التوثيق: `OpenAPI / Swagger`
- الكاش والمهام الخلفية: `Redis` لاحقًا عند الحاجة
- النشر: `Docker Compose` في البداية، ثم يمكن التوسع لاحقًا

## لماذا Spring Boot لهذا المشروع

هذا المشروع أقرب إلى ERP داخلي وليس CRUD بسيط. النظام يحتوي على:

- طلبيات بحالات متعددة.
- أسطر طلبية قابلة للتعديل مع حفظ الأثر.
- مخزون وحجوزات وحركات.
- زبائن وأرصدة ومدفوعات داخلة وخارجة.
- مشتريات وفواتير طريق ومرتجعات.
- صلاحيات دقيقة بين البائع والإدارة والمالك.
- طباعة وتقارير وإرسال PDF عبر واتساب.
- سجل عمليات Audit لكل تغيير حساس.

Spring Boot مناسب لهذا النوع لأنه قوي في:

- `Transactions`
- `Security`
- `Validation`
- `Service Layer`
- `Testing`
- التكامل مع PostgreSQL و Flyway
- بناء نظام طويل العمر ومنظم.

## أسلوب البناء المطلوب

لا يبدأ المشروع كـ microservices.
الاختيار الصحيح هو:

```text
Modular Monolith
```

أي مشروع Spring Boot واحد، لكنه مقسم داخليًا إلى modules واضحة. هذا يقلل التعقيد ويترك الباب مفتوحًا لفصل خدمات مستقلة لاحقًا مثل الطباعة أو WhatsApp أو AI.

## الطبقات القياسية

- `Controller`: استقبال HTTP فقط، بدون منطق عمل ثقيل.
- `Request DTO`: مدخلات الفرونت مع validation.
- `Response DTO`: مخرجات API للفرونت.
- `Service / UseCase`: منطق العمل الحقيقي.
- `Repository`: الوصول للبيانات.
- `Entity`: تمثيل الجداول.
- `Policy / Security`: التحقق من الصلاحيات.
- `Event`: إشعارات داخلية عند حدوث تغييرات مهمة.
- `Job`: مهام خلفية لاحقًا للطباعة، واتساب، AI.

## قواعد قاعدة البيانات

- PostgreSQL هو الاختيار الرسمي.
- لا يتم الاعتماد على `spring.jpa.hibernate.ddl-auto=update`.
- في التطوير والإنتاج يجب استعمال Flyway migrations.
- في الإنتاج يفضل:

```properties
spring.jpa.hibernate.ddl-auto=validate
```

- كل تعديل schema يكون migration جديد.
- لا يتم تعديل migration قديم بعد تطبيقه.
- يجب أخذ backup قبل migrations الكبيرة في الإنتاج.

## قواعد الـ Multi-Tenancy

- المنصة تستخدم schema ثابتًا باسم `platform`.
- كل مشترك يستخدم schema منفصلًا مثل `tenant_a1b2c3d4`.
- جداول الطلبيات، الزبائن، الأصناف، المخزون، المحاسبة، والإعدادات لا تنشأ في `public`.
- كل request تشغيلي يجب أن يمر عبر `TenantContext` يحدد schema المشترك من JWT أو subdomain أو header موثوق.
- لا يقبل الباك إند `schema_name` من جسم الطلب.
- خدمة `provisioning` مسؤولة عن إنشاء schema المشترك، تطبيق Flyway tenant migrations، seed البيانات الأساسية، وإنشاء أول admin.
- مستخدمو مالك المنصة منفصلون عن مستخدمي المشتركين: `platform.platform_users` مقابل `tenant_x.users`.
- أي وصول دعم من platform إلى بيانات tenant يجب أن يسجل في audit log.

## قواعد مهمة للدومين

### تجربة الصلاحيات والميزات

- الواجهة لا تخفي الميزات بسبب الصلاحيات أو الخطة. كل Route أو Action رئيسي يبقى ظاهرًا، لكن التنفيذ يمر عبر `AccessDecisionService`.
- `AccessDecisionService` يجب أن يفحص بالترتيب:
  1. حالة المشترك والاشتراك.
  2. هل الميزة مفعلة في خطة المشترك عبر `platform.plan_features`.
  3. صلاحيات المستخدم داخل tenant عبر RBAC.
  4. صلاحيات الشراكة إذا كان الإجراء يعتمد على Partner.
  5. نطاق البيانات مثل مالك الطلبية أو الولاية أو المخزن.
- نتيجة الفحص تكون DTO موحد:
  `allowed`, `denial_reason`, `required_feature`, `required_permission`, `required_partner_permission`, `message`, `action_type`, `action_url`.
- إذا كان المنع بسبب الخطة، ترجع API سبب `plan_required` وتعرض الواجهة Modal ترقية الخطة.
- إذا كان المنع بسبب صلاحية مستخدم داخل المشترك، ترجع `user_permission_required` وتعرض الواجهة رسالة "تواصل مع مالك الشركة أو الإدارة".
- إذا كان المنع بسبب صلاحية شريك، ترجع `partner_permission_required` وتعرض الواجهة رسالة "تواصل مع الشريك ليمنحك الصلاحية".
- كل API مهم يجب أن يعيد نفس قرار المنع، ولا يعتمد على الواجهة فقط.
- محاولات المنع تسجل في `access_denial_events` أو audit/event مناسب حتى يمكن تحليل الطلب على الصلاحيات والترقيات.

### الطلبيات

- البائع ينشئ طلبية بحالة `draft` أو `submitted`.
- الإدارة فقط تؤكد الطلبية إلى `confirmed`.
- السطور المحذوفة إداريًا لا تحذف فعليًا.
- كل تعديل في طلبية مؤكدة أو قيد المراجعة يجب أن ينتج سجل audit/event.

### المخزون

- لا تجعل رقم المخزون في جدول المنتجات هو الحقيقة الوحيدة.
- الحقيقة الأساسية تكون من:
  - `stock_movements`
  - `stock_reservations`
  - جدول summary سريع للقراءة مثل `product_stocks`
- تأكيد الطلبية وحجز المخزون يجب أن يتم داخل `@Transactional`.
- عند حجز كميات يجب استخدام locking مناسب في PostgreSQL لمنع بيع نفس الكمية مرتين.

### التصنيع

- وحدة `manufacturing` مسؤولة عن طلبات تصنيع المنتجات النهائية داخل شركة المشترك، سواء كانت مرتبطة ببيع أو بسد نقص مخزون أو بتصنيع خاص.
- إنشاء طلب تصنيع لا يضيف مخزونًا نهائيًا ولا يرحل تكلفة مباشرة؛ هو سجل تشغيلي ينتظر الاعتماد.
- `ManufacturingRequestService` يملك انتقالات الحالة، ولا يسمح بتغيير status من الواجهة مباشرة خارج مسار رسمي:
  `draft -> approved -> queued -> in_production -> quality_check -> ready_to_ship -> in_transit -> received`.
- الاعتماد يمكن أن يستدعي `InventoryReservationService` لحجز المواد الخام في `stock_reservations`، مع locking حتى لا تستهلك نفس المادة في طلبين.
- بدء التصنيع يسجل حركة صرف مواد خام أو تحويل WIP حسب إعدادات الشركة، ويكتب `manufacturing_events`.
- فحص الجودة يتم عبر `manufacturing_quality_checks`: مقبول، إعادة عمل، مرفوض. الكمية المرفوضة لا تدخل المخزون المتاح.
- الاستلام هو اللحظة الوحيدة التي تنشئ `stock_movement` دخول للمنتج النهائي في مخزن الوجهة.
- إذا كان `source_type = sold_order` ومرتبطًا بسطر طلبية بيع، بعد الاستلام يحجز النظام المنتج النهائي للطلبية أو يربط الكمية بها.
- إذا كان `source_type = stock_replenishment` تدخل الكمية كمخزون عام قابل للبيع حسب إعداد المخزن.
- العربون يحفظ في طلب التصنيع أو ledger المدفوعات حسب السياسة، لكنه لا يعني اكتمال الإنتاج. يمكن منعه من الانتقال إلى `approved` إذا كانت سياسة الشركة تتطلب دفعه.
- كل انتقال حالة يطلق event مثل `manufacturing.started` أو `manufacturing.received` لتغذية الإشعارات والتدقيق.
- أي إلغاء بعد صرف مواد يحتاج سياسة واضحة: إرجاع مواد غير مستهلكة، تسجيل هالك، أو تحويل الطلب إلى إعادة عمل. لا يسمح بتغيير الحالة فقط بدون أثر مخزني عند وجود حركات.
- عمليات التصنيع يجب أن تكون `@Transactional` وتستخدم idempotency key عند استقبال أحداث من واجهات أو وظائف خلفية حتى لا تتكرر حركات المخزون.

### المدفوعات ورصيد الزبون

- الرصيد لا يكون رقمًا يدويًا فقط.
- الأفضل بناء ledger:
  - opening balance
  - order debits
  - payment in
  - payment out
- payment allocations
- توزيع المدفوعات على الطلبيات يجب أن يكون منظمًا وقابلًا للتتبع.

### المحاسبة وشجرة الحسابات

- شجرة الحسابات لا تكون mock data؛ تزرع من `account_templates` عند إنشاء المشترك.
- الحسابات الرقابية مثل العملاء والموردين يجب أن تلزم `subledger_entity_id` في سطر القيد.
- لا يرحّل النظام على حساب أب أو حساب معطل أو حساب لا يسمح بالترحيل اليدوي.
- كل قيود العمليات التلقائية تولد عبر `accounting_rules` وليس if/else مبعثرة داخل خدمات البيع والشراء.
- دفاتر اليومية (`journal_books`) تحدد الترقيم، المصدر، وهل يحتاج القيد موافقة.
- `accounting_settings` يجب أن تفشل مبكرًا برسالة واضحة إذا كان أي حساب ربط ناقصًا.
- المخزون المحاسبي يعتمد على `inventory_valuation_layers` حتى يمكن حساب تكلفة البضاعة بدقة.
- التقارير المالية تعتمد على `report_section` و `cash_flow_section` داخل الحسابات، وليس على أسماء الحسابات.
- أي اختلاف بين حساب رقابي وتفصيل sub-ledger يظهر في تقرير مطابقة ولا يخفى.

### الشراكات وربط الموردين

- كود الربط يستخدم للتبادل بين المشتركين، لكن الربط التشغيلي الدائم يحفظ `partner_uuid` داخل علاقة الشراكة وداخل بطاقة المورد عند تأكيد المستخدم.
- عند إرسال أو قبول طلب ربط، تستدعي خدمة الشراكات `SupplierIdentityMatcher` داخل schema المشترك الحالي فقط.
- المطابقة لا تعتمد على الاسم، لأن الاسم قد يكتب بصيغ مختلفة. ترتيب الفحص: الرقم الضريبي، السجل التجاري، البريد، الهاتف.
- إذا وجد النظام مورداً واحداً فقط يطابق قيمة ثابتة، يرجع `supplier_link_suggestion` للواجهة ويولد حدث `partnership.supplier_match_found`.
- تعرض الواجهة سؤال تأكيد: هل هذا هو نفس المورد؟ عند الموافقة يحدث النظام بطاقة المورد (`linked_partner_uuid`, `link_match_method`, `link_confirmed_at`) داخل نفس tenant transaction ثم يكمل قبول الربط.
- إذا ظهرت أكثر من بطاقة بنفس القيمة، تسجل الحالة `ambiguous` ولا يحدث أي ربط تلقائي حتى يراجع المستخدم بطاقة المورد.
- كل تأكيد أو تجاهل للمطابقة يسجل في audit log حتى يمكن تتبع من ربط المورد وبأي حقل.

#### مزامنة فواتير/طلبيات البيع والشراء بين المشتركين

- أي تحويل بين فاتورة شراء وفاتورة بيع يجب أن يمر عبر `platform.partner_document_links`، ولا يسمح للفرونت أو خدمة tenant أن تكتب مباشرة داخل schema الطرف الآخر.
- عندما ينشئ مشترك أمر شراء لمورد مربوط، تتحقق خدمة الشراكات من `tenant_partnerships.status = active` ومن صلاحية `create_purchase_link`، ثم تنشئ مستند بيع مقابل عند الطرف المورّد بحالة `pending_partner_confirmation`.
- عندما ينشئ مشترك أمر بيع لزبون مربوط، تتحقق خدمة الشراكات من صلاحية `create_sales_link`، ثم تنشئ مستند شراء مقابل عند الطرف الزبون بحالة `pending_partner_confirmation`.
- كل مستند يحتفظ برقمه المحلي وحالته المحلية، ويرتبط فقط عبر `partner_document_link_public_id`.
- التزامن يستخدم `idempotency_key` حتى لا تتكرر الفاتورة المقابلة عند إعادة المحاولة أو تكرار الطلب.
- التعديلات اللاحقة ترسل أحداثًا مثل `partner_document.status_changed` أو `partner_document.quantity_changed` بدل تعديل الطرف الآخر بصمت.
- لا توجد كتابة مباشرة من خدمة Tenant في مخزون Tenant آخر، لكن حركة المخزون تتم تلقائياً داخل كل طرف حسب مرحلة المستند.
- طرف البيع يحجز الكمية عند القبول إذا كانت السياسة تتطلب الحجز، ثم ينشئ `stock_movement` خروج عند الشحن أو تأكيد خروج البضاعة.
- طرف الشراء يسجل الكمية كمنتظرة أو قيد الطريق، ثم ينشئ `stock_movement` دخول عند تأكيد الاستلام فقط.
- القبول الجزئي والشحن الجزئي والاستلام الجزئي يجب أن تنعكس على الكميات في السطور، ولا يسمح بإدخال كمية أكبر من المشحون/المقبول.
- القيود المحاسبية محلية أيضاً: البائع يرحل المبيعات وتكلفة البضاعة حسب قواعده، والمشتري يرحل المخزون وذمم المورد عند الاستلام أو اعتماد فاتورة المورد.
- أي رفض أو قبول جزئي أو إلغاء يجب أن يحدث المستند المقابل بحالة واضحة، ويولد notification للطرف الآخر.
- الإلغاء قبل الشحن يحرر الحجز عند البائع ويلغي الوارد المنتظر عند المشتري.
- الإلغاء بعد الشحن يتحول إلى `partner_document_claim` لأن البضاعة خرجت من مخزون البائع؛ لا يسمح بإرجاع الكميات بمجرد تغيير status.
- المرتجع بعد الاستلام ينشئ حركة خروج عند المشتري وحركة دخول عند البائع عند وصول المرتجع، وقد يدخل إلى مخزون متاح أو حجر/تالف حسب نتيجة الفحص.
- النقص أو الضياع أو التلف يسجل كمطالبة مرتبطة بالمستند الأصلي. المشتري يستلم فقط الكمية الفعلية، والفرق يبقى في claim حتى يحسم بتعويض، إعادة شحن، إشعار دائن، رد مبلغ، أو تحمل خسارة.
- كل claim يجب أن يملك سطور كميات مستقلة وأحداثاً قابلة للتدقيق، ولا يغير سطور البيع/الشراء الأصلية إلا عبر action صريح.

### الإشعارات

- لا تنشئ الواجهة إشعارات مباشرة. أي عملية مهمة تطلق Domain Event مثل `product.price_changed` أو `inventory.low_stock` أو `partnership.requested`.
- وحدة `notifications` تستقبل الأحداث وتطبق `notification_rules` لتحديد النوع، الشدة، المستلمين، والقنوات.
- إشعارات مالك المنصة تستخدم جداول `platform.platform_notifications`، وإشعارات المشترك اليومية تستخدم جداول tenant schema.
- إشعارات الربط بين مشتركين تمر عبر `platform` ثم تعرض للمشتركين حسب `recipient_tenant_id` و `tenant_user_public_id` أو تنسخ كإشعار محلي مرتبط بـ `platform_notification_id`.
- حالة القراءة والأرشفة تكون لكل مستلم في `notification_recipients`، وليست على سجل الإشعار نفسه.
- يجب أن يحسب `RecipientResolver` سبب وصول الإشعار ويخزنه في `recipient_reason`.
- يجب دعم `dedupe_key` حتى لا يتكرر تنبيه نفس الصنف منخفض المخزون مرات كثيرة خلال نافذة قصيرة.
- يجب دعم `group_key` لتجميع أحداث متشابهة في إشعار واحد قابل للفتح.
- يجب تسجيل إجراءات المستخدم في `notification_actions` بدل الاكتفاء بتغيير `is_read`.
- يجب دعم `snoozed_until` و worker للتصعيد عند عدم المعالجة ضمن السياسة.
- يجب دعم outbox للتسليم غير المتزامن عبر in-app أولًا ثم email/WhatsApp/push لاحقًا.
- يجب أن تعمل قوالب الإشعارات حسب اللغة والقناة، ولا تخزن النصوص فقط داخل الكود.
- يجب أن تطبق سياسات الاحتفاظ والتنظيف بدون حذف audit log المرتبط.
- إشعارات الأمان والصلاحيات لا تكتم من المستخدم العادي؛ السياسة تحددها الإدارة أو مالك المنصة.

### الدعم والشات بين المالك والمشتركين

- وحدة `support` مسؤولة عن تذاكر الدعم والمحادثات والمرفقات بين tenant ومالك المنصة.
- التذاكر تحفظ في `platform.support_tickets` لأنها عابرة للـ tenant، لكن الرسائل لا تمنح فريق الدعم وصولاً مفتوحاً إلى بيانات tenant.
- فتح تذكرة يطلق event `support.ticket_opened` ويولد إشعاراً لفريق الدعم/المالك.
- إرسال رسالة يطلق `support.message_sent`: إذا المرسل tenant يصل إشعار للمالك/الدعم، وإذا المرسل platform يصل إشعار للمشترك صاحب التذكرة.
- حالة التذكرة لا تعدل يدوياً إلى `waiting_owner` أو `waiting_tenant`. هذه الحالات يحسبها النظام من آخر رسالة: رسالة tenant تجعلها `waiting_owner`، ورسالة platform تجعلها `waiting_tenant`.
- أفعال المستخدمين المسموحة على الحالة هي: فتح، إغلاق، إعادة فتح فقط.
- عند الإغلاق يمنع إرسال رسائل جديدة حتى يعاد فتح التذكرة.
- يجب دعم رسائل `text`, `image`, `audio`, `file`. الملفات تحفظ عبر storage service، والقاعدة تحفظ `storage_key` والـ metadata فقط.
- التسجيل الصوتي يحفظ كملف مرفق مع `duration_seconds` و `mime_type`، ولا يحفظ كنص base64 داخل قاعدة البيانات.
- حالة القراءة تكون عبر `support_ticket_participants.last_read_message_id` حتى تظهر badges للطرفين.
- إغلاق التذكرة لا يحذف الرسائل أو المرفقات؛ تطبق سياسة retention منفصلة، وأي حذف منطقي يجب أن يسجل في audit log.
- WebSocket/SSE لاحقاً يستخدم لتحديث الشات فورياً، لكن REST يجب أن يبقى المصدر الرسمي للإنشاء والحفظ.

### قفل التحرير

- وحدة `locking` مسؤولة عن أقفال التحرير القصيرة على الموارد الحساسة.
- لا تستخدم transaction طويلة في قاعدة البيانات لتغطية وقت بقاء المستخدم داخل شاشة التعديل.
- عند فتح شاشة تعديل، يستدعي الفرونت `acquire lock` حسب `resource_type` و `resource_id` و `lock_scope`.
- إذا يوجد lock فعال لمستخدم آخر، ترجع API معلومات القفل للواجهة حتى تعرض اسم المستخدم وتدخل وضع قراءة فقط.
- كل شاشة تعديل ترسل `heartbeat` دوريًا. إذا توقف heartbeat، ينتهي القفل عبر worker.
- كل endpoint يغير بيانات حساسة يجب أن يتحقق من `lock_token` عندما تكون سياسة المورد تتطلب قفلًا.
- عند الحفظ، يجب أيضًا التحقق من `version` أو `updated_at` لمنع overwrite حتى لو كان lock موجودًا.
- `take over` مسموح فقط بصلاحية واضحة ويسجل في audit log ويولد notification للمستخدم الأول.
- الموارد الأولى التي تحتاج قفلًا: طلبيات البيع، أوامر الشراء، الجرد والتسويات، الأصناف والأسعار، القيود اليدوية، إعدادات المحاسبة، صلاحيات المستخدمين، وشبكة الشركاء.

### فواتير الطريق

- فاتورة الطريق قد ترتبط بطلبية واحدة أو عدة طلبيات.
- اختيار عدة طلبيات يجب أن يتحقق من الولاية.
- الزبون التلقائي حسب الولاية يأتي من الإعدادات.
- دمج الأصناف يجب أن يحفظ أثر الأصناف الأصلية أو تفاصيل المصدر.

## تعليمات مهمة لـ Claude Code عند بدء الباك اند

عند إضافة مشروع الباك اند لاحقًا:

1. لا تستخدم `ddl-auto=update`.
2. أنشئ migrations عبر Flyway بمسارين: platform و tenant.
3. ابدأ بـ `platform.tenants/plans/subscriptions` ثم tenant identity/RBAC.
4. لا تبني CRUD عشوائي قبل تثبيت `TenantContext` والدومين والحالات.
5. استخدم DTOs ولا تعرض Entities مباشرة للفرونت.
6. أي عملية مالية أو مخزنية يجب أن تكون داخل `@Transactional`.
7. أي عملية مهمة يجب أن تطلق Domain Event قابلًا للتحويل إلى audit log أو notification.
8. أي تعديل حساس يجب أن يكتب audit log.
9. حافظ على modular monolith ولا تبدأ microservices.
10. اربط React الحالي تدريجيًا ولا تعيد بناء الواجهة من الصفر.
