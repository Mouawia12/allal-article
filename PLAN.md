## أولاً: الوحدات الجوهرية

### 1. الطلبيات (Orders)
**الحالة:** ✅ واجهة منجزة

- قائمة الطلبيات: جدول احترافي بتبويبات الحالة، بحث، فلترة بالولاية وحالة الدفع
- تفاصيل الطلبية:
  - عرض شامل لمعلومات الطلبية والأسطر
  - حالات الأسطر: pending / approved / modified / shipped / cancelled
  - لا توجد حالة "محذوف"؛ حذف السطر الإداري يتحول إلى `cancelled` مع كمية ملغاة مساوية للكمية المطلوبة
  - الأسطر الملغاة تبقى مرئية داخل الفاتورة بالخلفية الحمراء وخط العبور، لكنها لا تظهر في الطباعة
  - **أزرار الإجراء:** إرسال للإدارة | سجلها قيد مراجعة | تأكيد | إلغاء قبل الشحن | رفض | تعديل الفاتورة | **تحويل إلى فاتورة طريق** | **إنشاء مرتجع** | تسجيل الشحن | اكتمال طلبية
  - بعد الشحن يظهر زر "اكتمال طلبية"، وإذا مرّت 3 أيام على الشحن ولم تكتمل يغيرها النظام تلقائياً إلى مكتملة
  - زر إلغاء الطلبية يظهر حتى حالة مؤكدة، ويختفي نهائياً بعد الشحن
- إضافة طلبية — **واجهتان حسب دور المستخدم:**
  - **واجهة البائع (الحالية):** شبكة منتجات + سلة + نافذة كمية + بحث + فلترة
  - **واجهة الإداري (Odoo-style):** سطور نصية — يكتب اسم/كود الصنف فتنبثق نتائج فلترة فورية، يضيف الكمية ثم ينزل سطر جديد تلقائياً
  - كلا الواجهتين: اختيار قائمة أسعار + عمود سعر الصنف + عمود وزن الصنف + إجمالي وزن وإجمالي مبلغ الطلبية
- **عمود العلب/الكراطين في السلة:** يحسب تلقائياً (كمية ÷ عدد القطعة في العلبة)
- **بطاقة الزبون عند الاختيار:** لما يُختار زبون في إضافة طلبية يتحول زر "إضافة زبون" إلى أيقونة عين — الضغط عليها يفتح بطاقة منبثقة بجميع بيانات الزبون (اسم، هاتف، ولاية، رصيد، آخر طلبيات)
- **مرتجع الطلبية:** زر "إنشاء مرتجع" يفتح نموذج لكل صنف (كمية المرتجع ≤ الكمية المعتمدة) + اختيار السائق + اختيار من استلمه في الإدارة
- **تحويل طلبيات متعددة:** من قائمة الطلبيات — تحديد أكثر من طلبية + زر "تحويل لفاتورة طريق"
- **قوائم الأسعار داخل الطلبية:** عند اختيار قائمة مثل "أسعار أحمد" أو "أسعار الجملة"، يتم تسعير الأصناف الموجودة في القائمة حسبها، وأي صنف غير موجود أو سعره داخل القائمة `0` يأخذ السعر الرئيسي من بطاقة الصنف.

**سير الحالات:**
```
مسودة → مرسلة للإدارة → قيد المراجعة → مؤكدة → مشحونة → مكتملة
                                  ↓
                              ملغاة / مرفوضة
```

**قواعد التنفيذ المطلوبة للباك إند:**
- `draft`: زر الإجراء الأساسي **إرسال للإدارة** فقط.
- `submitted`: زر الإجراء الأساسي **سجلها قيد مراجعة**.
- `under_review`: زر الإجراء الأساسي **تأكيد**، ويمكن رفض الطلبية.
- `confirmed`: يظهر زر **تسجيل الشحن**، ويظهر زر **إلغاء الطلبية** ما دامت لم تشحن بعد.
- `shipped`: يختفي زر الإلغاء والتعديل، ويظهر زر **اكتمال طلبية** وزر **إنشاء مرتجع**.
- `completed`: حالة إغلاق تشغيلي بعد الشحن، ولا تسمح بتعديل السطور؛ المرتجع يبقى ممكناً إذا وجدت كميات مشحونة.
- إذا مرّت 3 أيام على `shipped_at` والطلبية ما زالت `shipped`، يحولها النظام تلقائياً إلى `completed` ويكتب حدث `order.auto_completed`.
- زر **تحويل إلى فاتورة طريق** لا يظهر قبل `confirmed`، ويبقى متاحاً في `confirmed`, `shipped`, `completed`.
- عمود **شحن؟** يعرض `—` قبل تنفيذ الشحن، ويتحول إلى `✓` فقط عندما تصبح `shipped_qty > 0`.
- تعديل الفاتورة قبل الشحن:
  - تعديل `approved_qty` يحسب `cancelled_qty = requested_qty - approved_qty`.
  - تعديل `cancelled_qty` يحسب `approved_qty = requested_qty - cancelled_qty`.
  - أي سطر تم لمسه يصبح `modified`، والسطر غير الملموس عند الحفظ يصبح `approved`.
  - إذا `cancelled_qty = requested_qty` يصبح السطر `cancelled`، ويظهر مشطوباً في الواجهة ولا يظهر في الطباعة.
  - حذف السطر الإداري لا يحذف السطر من قاعدة البيانات، بل يحوله إلى `cancelled`.
- الشحن يتم كاملاً للكمية المعتمدة لكل سطر غير ملغى: `shipped_qty = approved_qty` و`line_status = shipped`.
- كل تغيير حالة أو كمية يجب أن يكتب `order_events` أو `order_item_events` باسم المستخدم وما الذي تغير.

### 2. الأصناف / المنتجات (Products)
**الحالة:** ✅ واجهة منجزة

- قائمة بعرض Grid/List + مقاييس المخزون
- مفضلة الأصناف:
  - زر نجمة في كروت وجدول الأصناف لإضافة/إزالة الصنف من مفضلة المستخدم
  - فلتر سريع باسم **المفضلة** داخل قائمة الأصناف وواجهة إضافة الطلبية السريعة
  - عداد للمفضلة ضمن مقاييس صفحة الأصناف
- تفاصيل الصنف:
  - زر واضح لإضافة/إزالة الصنف من المفضلة
  - 5 مقاييس مخزون: متوفر، محجوز، متاح، قيد الطلب، متوقع
  - **تاب "عمليات"** مع حقلين:
    - حقل كمية → يعرض الوزن الإجمالي + عدد العلب/الكراطين
    - حقل وزن → يقدر الكمية المقابلة
    - حقل التعليب: عدد القطع في كل علبة/كرطون
  - تاب سجل الأسعار
  - تاب حركات المخزون
  - تاب الطلبيات المرتبطة
- إضافة/تعديل صنف: نموذج شامل + رفع صورة + استيراد بالذكاء الاصطناعي
- قوائم الأسعار:
  - إنشاء أكثر من قائمة أسعار باسم واضح مثل "أسعار أحمد"، "أسعار الجملة"، "أسعار نصف جملة"
  - القائمة لا يلزم أن تحتوي كل الأصناف؛ الأصناف غير المسعّرة داخلها تستخدم السعر الرئيسي في بطاقة الصنف
  - قيمة سعر `0` داخل القائمة تعتبر غير مفعلة لهذا الصنف وليست بيعاً مجانياً، فيرجع النظام للسعر الرئيسي
  - صفحة الطلبية تعرض قائمة الأسعار المختارة، وكل سطر يعرض السعر المستخدم ومصدره: قائمة أسعار أو السعر الرئيسي

### 3. الزبائن (Customers)
**الحالة:** ✅ واجهة منجزة (محدّثة)

**بيانات الزبون:**
- اسم، هاتف رئيسي، هاتف ثاني، **الولاية (قائمة 58 ولاية)**, عنوان تفصيلي
- مسار الشحن، البائع المسؤول، **الرصيد الافتتاحي** (رصيد قبل البرنامج)

**بطاقة الزبون:**
- تبويبات: البيانات | سجل الطلبيات | سجل الدفعات | الشحن
- شريط ملخص الرصيد: إجمالي الطلبيات | المدفوع | الرصيد الافتتاحي | **الرصيد المتبقي**

**سجل الطلبيات:**
- عرض ملون: مدفوعة (أخضر) | مدفوعة جزئياً (برتقالي) | غير مدفوعة (أحمر)
- منطق الرصيد: يوزع المدفوعات على الطلبيات من الأقدم للأحدث
- طباعة كشف الحساب + **إرسال عبر واتساب PDF**

**سجل الدفعات:**
- **استلام دفعة من الزبون:** المبلغ + طريقة الدفع (نقدي/بنكي/شيك) + من استلم + من دفع
- **إرجاع مبلغ للزبون:** نفس الحقول (حالة العكس — أحياناً الزبون يدفع زائداً)
- عرض كل دفعة بلون حسب الاتجاه (أخضر: داخل / أحمر: خارج)

**تاب الشحن:**
- مسار الشحن + سجل الشحنات المرتبطة بهذا الزبون

**فلترة الزبائن:**
- بالولاية (قائمة 58 ولاية) + بحث نصي + تبويبات: الكل | نشطون | غير نشطين | **مديونون**

**تاب المديونون:**
- يعرض فقط الزبائن الذين رصيدهم المتبقي > 0 (أي: إجمالي الطلبيات − المدفوع − الرصيد الافتتاحي > 0)
- فلتر الولاية يُطبَّق فوق هذا التاب كما بقية التبويبات
- اسم التاب يظهر بالأحمر للتمييز البصري

**طباعة التقرير الذكية:**
- زر "طباعة تقرير" في شريط الأدوات جانب "إضافة زبون"
- الزر يتكيّف حسب التاب النشط:
  - في تاب **المديونون**: الزر يصبح "طباعة تقرير الديون"
  - في بقية التبويبات: يبقى "طباعة تقرير"
- عند الطباعة في تاب المديونون → **قالب الديون التفصيلي**: جدول بكل زبون مدين مع (الرصيد الافتتاحي / إجمالي الطلبيات / المدفوع / الرصيد المتبقي) + قائمة طلبياته غير المدفوعة أو المدفوعة جزئياً (رقم الطلبية، التاريخ، المبلغ، المدفوع، المتبقي)
- عند الطباعة في بقية التبويبات → **القالب العادي**: جدول مبسط (الاسم، الهاتف، الولاية، مسار الشحن، البائع المسؤول، الحالة)
- كلا القالبين يتضمنان: عنوان الشركة، التاريخ، الولاية المختارة (إن وُجدت)، إجمالي السجلات

### 4. المخزون (Inventory)
**الحالة:** ✅ واجهة منجزة

- جدول تفصيلي بمقاييس: متوفر، محجوز، طلبيات غير مؤكدة، متاح، متوقع، حد التنبيه
- شريط تقدم للاستهلاك
- تبويبات: الكل | نفذ | منخفض | متوفر
- لون الصف حسب الحالة
- دعم تعدد المستودعات: فلتر مستودع + عرض الرصيد حسب `product_id + warehouse_id`
- تبويب **إدارة المستودعات**: بطاقة لكل مستودع تعرض المدينة، النوع، المسؤول، عدد الأصناف، المتاح، المحجوز، ونسبة استخدام الطاقة
- تبويب **تحويلات المخزون**: سجل تحويلات بين المستودعات مع رقم حركة، المصدر، الوجهة، الصنف/النطاق، الكمية، المستخدم، والسبب
- تحويل صنف بين مستودعين: ينقل الكمية المتاحة فقط ولا يلمس الكمية المحجوزة
- تحويل مستودع كامل إلى مستودع آخر: ينقل كل الكميات المتاحة لكل الأصناف، ويترك المحجوز في المستودع المصدر حتى لا تنكسر الطلبيات المؤكدة
- كل تحويل يجب أن يكتب `stock_transfers` + `stock_transfer_items` + حركتي مخزون لكل سطر: خروج من المصدر ودخول للوجهة
- لا يسمح بتحويل إلى نفس المستودع، ولا بتحويل كمية أكبر من `available_qty`

### 5. التصنيع (Manufacturing) — **جديد**
**الحالة:** ✅ واجهة Mock + خطة + Schema، 🔄 ربط Backend لاحقًا

**الفكرة:** بعض المشتركين عندهم مصانع أو ورش داخلية. النظام يجب أن يسمح بفتح طلب تصنيع لصنف معين، ثم يتابعه مسؤولو التصنيع من الاعتماد، طابور المصنع، الإنتاج، الجودة، الشحن الداخلي، إلى أن يصل للمقر أو المخزن الذي طلبه.

#### واجهة التصنيع

- صفحة جديدة: `src/layouts/manufacturing/index.js`
- رابط الصفحة: `/manufacturing`
- تظهر في السايدبار باسم **التصنيع** بين المخزون والمشتريات.
- الصفحة تعرض:
  - مؤشرات: طلبات نشطة، قيد التصنيع، جاهزة للتسليم، عربون متبقٍ، ومتأخرة.
  - جدول طلبات تصنيع مع فلترة حسب الحالة والبحث برقم الطلب/الصنف/المصنع/الفرع.
  - بطاقة تفاصيل للطلب المختار: الصنف، الكمية، السبب، المقر المستلم، المصنع، خط الإنتاج، المسؤول، المواد، الجودة، العربون، وخط المتابعة.
  - زر "طلب تصنيع جديد" بنموذج يحدد الصنف، الكمية، سبب التصنيع، المخزن المستلم، المصنع، خط الإنتاج، المسؤول، الأولوية، تاريخ التسليم، العربون، والملاحظات.
  - أزرار تحديث الحالة حسب المرحلة الحالية فقط.

#### أسباب طلب التصنيع

- `sold_order`: مرتبط ببيع أو طلبية زبون. يستخدم عندما تكون الكمية مصنعة خصيصًا لتلبية طلب مؤكد أو شبه مؤكد.
- `stock_replenishment`: سد نقص مخزون. يستخدم عندما يفتح المخزون أو النظام طلبًا بسبب نزول المتاح تحت حد التنبيه.
- `custom_order`: تصنيع خاص بمواصفات غير قياسية، غالبًا يحتاج ملاحظات، مرفقات، واعتماد مقاسات.

#### دورة حياة طلب التصنيع

- `draft`: مسودة أو طلب يحتاج عربون/اعتماد.
- `approved`: معتمد من الإدارة أو المسؤول.
- `queued`: دخل طابور المصنع.
- `in_production`: بدأ التصنيع وسحب/حجز المواد.
- `quality_check`: فحص جودة، مع دعم مقبول/إعادة عمل/مرفوض.
- `ready_to_ship`: جاهز للشحن الداخلي.
- `in_transit`: في الطريق إلى المقر أو المخزن المستلم.
- `received`: تم استلام المنتج النهائي ودخل المخزون.
- `cancelled`: ملغى مع سبب وتدقيق.

#### قواعد المخزون والتكلفة

- إنشاء طلب التصنيع لا يزيد مخزون المنتج النهائي.
- الاعتماد يمكن أن ينشئ حجزًا للمواد الخام أو يمنع استعمالها في أوامر أخرى.
- عند بدء التصنيع، تسجل حركة صرف مواد خام من مخزن المواد أو تحويلها إلى WIP حسب إعدادات الحسابات.
- عند الاستلام، ينشئ النظام حركة دخول للمنتج النهائي في مخزن الوجهة فقط، وتصبح الكمية متاحة حسب سياسة المخزن.
- إذا كان الطلب مرتبطًا ببيع، يمكن حجز المنتج النهائي مباشرة للطلبية بعد الاستلام.
- إذا كان الطلب لسد نقص مخزون، يدخل المنتج النهائي كمخزون عام قابل للبيع.
- التالف أو المرفوض في الجودة لا يدخل مخزونًا متاحًا؛ يدخل إعادة عمل أو مخزن تالف/حجر.
- التكلفة تجمع مواد خام + أجور تشغيل + مصاريف تصنيع غير مباشرة، ثم تفتح طبقة تقييم للمنتج النهائي.

#### العربون والملاحظات

- العربون اختياري، ومهم في التصنيع الخاص أو طلبية زبون قبل بدء الإنتاج.
- حالة العربون: بدون عربون، منتظر، جزئي، مدفوع.
- العربون لا يعني اكتمال التصنيع، لكنه قد يمنع الانتقال من `draft` إلى `approved` حسب سياسة الشركة.
- الملاحظات تحفظ سبب الطلب، مواصفات الزبون، شروط التسليم، وأي ملاحظات تصنيع.

#### صلاحيات وأدوار

- طالب التصنيع يفتح الطلب ويتابع الاستلام.
- مسؤول التصنيع يغير حالات المصنع والإنتاج.
- فاحص الجودة يسجل القبول أو إعادة العمل أو الرفض.
- مسؤول المخزن يؤكد الاستلام النهائي ويدخل الكمية للمخزون.
- كل إجراء يسجل في `manufacturing_events` ويولد إشعارًا حسب الدور.

### 6. المشتريات (Purchases) — **جديد**
**الحالة:** ✅ واجهة منجزة

- قائمة أوامر الشراء: جدول مع حالة الأمر + حالة الدفع + المستلِم
- تبويبات: الكل | في الانتظار | مؤكد | مستلم | ملغى
- إحصائيات: غير مدفوع | مؤكد | مستلم | مرتجع مشتريات + إجمالي الديون للموردين
- صفحة تفاصيل أمر الشراء + إضافة/تعديل أمر شراء
- أوامر الشراء تدعم اختيار قائمة أسعار/تكاليف عند الإنشاء، مع عمود سعر لكل صنف وملخص إجمالي أمر الشراء
- إذا لم يكن للصنف سعر داخل قائمة الشراء المختارة أو كان السعر `0`، يستخدم النظام السعر الرئيسي المعرّف في بطاقة الصنف كقيمة fallback
- مرتجع مشتريات: زر داخل صفحة فاتورة/أمر الشراء يفتح نموذجاً مفصلاً لكل صنف، ولا يسمح بكمية أكبر من الكمية المستلمة غير المرتجعة
- أسطر المشتريات تعرض `المستلم` + `المرتجع` + `الصافي`، مع تلوين الصنف المرتجع جزئياً بلون تحذيري والصنف المرتجع بالكامل بالأحمر
- المرتجع ينعكس على المخزون والذمم والضريبة: خروج مخزون، تخفيض ذمة المورد أو إشعار دائن، وعكس TVA القابل للاسترجاع
- بعد تثبيت واجهات المشتريات، نقطة العمل التالية في البرنامج هي ربط مرتجع المشتريات بخدمة Backend وقيود محاسبية فعلية
- صفحة الموردين: بطاقات موردين، أرصدة، أوامر شراء، دفعات، وتعديل بيانات الربط
- ربط المورد بشريك: معرف الشريك `UUID` هو الرابط الثابت بين المشتركين، وليس الاسم
- عند قبول طلب ربط من مشترك آخر، يفحص النظام بطاقات الموردين الموجودة عبر بيانات ثابتة وغير متكررة: الرقم الضريبي، السجل التجاري، البريد، الهاتف
- إذا وجد النظام تطابقاً فريداً، تعرض الواجهة تنبيهًا: "هل هذا هو نفس المورد؟"؛ عند التأكيد يربط بطاقة المورد تلقائياً بدون أن يذهب المستخدم لصفحة الموردين ويعدلها يدوياً
- إذا كان التطابق غامضاً أو مكرراً، لا يتم الربط تلقائياً ويطلب النظام مراجعة يدوية

#### مزامنة فواتير/طلبيات الشركاء المرتبطين

**الفلسفة:** عندما يكون مشتركين مرتبطين بصلاحية تبادل الطلبيات، لا يكتب أي طرف مباشرة داخل قاعدة الطرف الآخر من الفرونت. الباك إند ينشئ مستندًا مقابلاً عند الطرف الثاني عبر خدمة مزامنة رسمية، ويظل كل طرف يراجع ويؤكد ويشحن/يستلم حسب إجراءاته.

**من شراء إلى بيع:**
1. المشترك A ينشئ فاتورة/أمر شراء ويختار مورداً مربوطاً بمشترك B.
2. النظام ينشئ عند B فاتورة/طلبية بيع واردة من الشريك بحالة `pending_partner_confirmation`.
3. B يراجع الأصناف والأسعار والكميات، ثم يؤكد أو يرفض أو يعدل الكميات حسب المخزون والصلاحيات.
4. عند تأكيد B، تظهر الحالة عند A أن طلب الشراء تم قبوله أو قبوله جزئياً أو رفضه.
5. الشحن عند B والاستلام عند A يبقيان خطوتين منفصلتين؛ لا يصبح الشراء مستلماً عند A إلا بعد تأكيد الاستلام محلياً.

**من بيع إلى شراء:**
1. المشترك B ينشئ فاتورة/طلبية بيع ويختار زبوناً مربوطاً بمشترك A.
2. النظام ينشئ عند A فاتورة/أمر شراء وارد من الشريك بحالة `pending_partner_confirmation`.
3. A يراجع أمر الشراء ويؤكده أو يرفضه.
4. بعد قبول A، يكمل B مسار التأكيد والحجز والشحن، ثم ينعكس تقدم الشحن إلى A كحالة توريد منتظرة.

**قواعد الربط:**
- لا تتم المزامنة إلا إذا كانت الشراكة `active` والصلاحية المناسبة مفعلة في `permissions_json`: صلاحية `create_purchase_link` لمسار شراء → بيع، وصلاحية `create_sales_link` لمسار بيع → شراء.
- الربط يعتمد على `partner_document_link_public_id` كمعرف مشترك بين المستندين، وليس على رقم الفاتورة المحلي.
- كل طرف يحتفظ برقمه المحلي: رقم أمر الشراء عند المشتري ورقم أمر البيع عند البائع.
- أي تعديل بعد الإنشاء يولد حدث مزامنة جديد، ولا يغيّر المستند المقابل بصمت.
- لا توجد كتابة مباشرة من Tenant إلى مخزون Tenant آخر، لكن حركة المخزون نفسها تتم تلقائياً داخل كل طرف حسب المرحلة.

**قواعد حركة المخزون:**
- عند قبول البائع للطلبية المقابلة، يحجز النظام الكمية من مخزون البائع إذا كانت سياسة البيع تتطلب الحجز.
- عند الشحن أو تأكيد خروج البضاعة، ينشئ النظام حركة خروج من مخزون البائع.
- عند المشتري، يظهر المستند ككمية منتظرة أو وارد قيد الطريق، ولا تدخل الكمية في المخزون المتاح للبيع بعد.
- عند تأكيد الاستلام عند المشتري، ينشئ النظام حركة دخول للمخزون وتصبح الكمية متاحة حسب إعداد المستودع.
- إذا كان القبول جزئياً، تخصم وتدخل فقط الكميات المقبولة/المشحونة/المستلمة.
- القيود المحاسبية أيضاً محلية: البائع يرحل البيع وتكلفة البضاعة عند مرحلته، والمشتري يرحل المخزون/ذمم المورد عند الاستلام أو اعتماد فاتورة المورد.

**الإلغاء والمرتجع والضياع بين مشتركين:**
- الإلغاء قبل قبول البائع: يلغي المستندين بدون حركة مخزون.
- الإلغاء بعد الحجز وقبل الشحن: يحرر حجز مخزون البائع، ويلغي الوارد المنتظر عند المشتري.
- الإلغاء بعد الشحن لا يعامل كإلغاء عادي؛ يتحول إلى مطالبة `claim` أو مرتجع لأن مخزون البائع خرج فعلاً.
- المرتجع بعد الاستلام: المشتري ينشئ مرتجعاً مرتبطاً بالمستند، تخرج الكمية من مخزونه أو مخزن المرتجعات، والبائع يدخلها عند الاستلام إلى مخزون متاح أو حجر/تالف حسب حالة الصنف.
- إذا وصل جزء من الكمية فقط، يؤكد المشتري استلام الكمية الفعلية ويفتح مطالبة نقص/ضياع للفرق.
- الصنف الضائع في الطريق يبقى غير داخل لمخزون المشتري، ومخزون البائع يبقى خارجاً؛ يحسم الحل عبر مطالبة: إعادة شحن بديل، إشعار دائن/خصم، رد مبلغ، أو تحمل خسارة/شركة الشحن.
- الصنف التالف يدخل عند المستلم إلى مخزن حجر/تالف وليس إلى المخزون المتاح حتى تقرر الإدارة إرجاعه أو قبوله بسعر مخفض أو إتلافه.
- كل مطالبة لها حالة وسجل أحداث، ولا تعدل مستند الطرف الآخر بصمت.

### 7. فواتير الطريق (Road Invoices) — **جديد**
**الحالة:** ✅ واجهة منجزة

**مفهوم فاتورة الطريق:** وثيقة شحن تُعطى للسائق وتحتوي على أصناف من طلبية واحدة أو أكثر موجهة لنفس الولاية.

**إنشاء فاتورة طريق (3 طرق):**
1. من صفحة تفاصيل الطلبية: زر "تحويل إلى فاتورة طريق"
2. من قائمة الطلبيات: تحديد أكثر من طلبية + زر "تحويل لفاتورة طريق"
3. من صفحة فواتير الطريق: زر "فاتورة جديدة" أو "تحويل من طلبيات" (يفتح نافذة بحث واختيار)

**محتوى فاتورة الطريق:**
- الولاية + الزبون (تلقائي من الإعدادات، قابل للتغيير) + السائق + التاريخ
- جدول الأصناف القابل للتعديل: حذف صنف | تعديل الكمية | تعديل السعر
- عمود الوزن لكل صنف + الوزن الإجمالي
- **زر "دمج الأصناف":** يجمع الأصناف من نفس الفئة (يجمع الكميات في الصنف الأول، يحذف الباقي) — الهدف: توفير الورق عند الطباعة

**عند اختيار أكثر من طلبية:**
- إذا كانت من ولاية واحدة → الزبون التلقائي من الإعدادات
- إذا كانت من ولايات مختلفة → تحذير

**قائمة فواتير الطريق:**
- جدول: رقم | التاريخ | الولاية | الزبون | السائق | الطلبيات | الأصناف | الوزن | عدد الطباعة | الحالة
- فلترة بالولاية
- أزرار: عرض | طباعة | **إرسال واتساب PDF**
- تبويبات: الكل | مسودات | في الطريق | مسلّمة

### 7. المحاسبة (Accounting)
**الحالة:** ✅ واجهة منجزة (Frontend)

**الفلسفة:** محاسبة قيد مزدوج كاملة (Double-Entry) مرتبطة بكل عمليات النظام (بيع، شراء، دفع، مرتجع).

**مرجع التصميم الاحترافي:** [ACCOUNTING_PROFESSIONAL_BLUEPRINT.md](/Users/mw/Downloads/allal-article/ACCOUNTING_PROFESSIONAL_BLUEPRINT.md:1)

#### لوحة المحاسبة (`/accounting/dashboard`) — **مطلوبة**
- ملخص أرصدة: الصندوق، البنك، العملاء، الموردين، المخزون، المبيعات، المصاريف
- تنبيهات: قيود غير متوازنة، قيود مسودة، حسابات ربط ناقصة، فترات مفتوحة، حسابات رقابية بدون sub-ledger
- اختصارات: قيد جديد، ربط الحسابات، الأرصدة الافتتاحية، مطابقة الذمم

#### قوالب شجرة الحسابات (`/accounting/chart-templates`) — **مطلوبة**
- اختيار قالب أولي عند إنشاء المشترك: `SCF Algeria Trading`
- اعتماد نظام كود رقمي مثل `1000`, `1001`, `1010`:
  - 4 خانات افتراضياً.
  - أرقام فقط بدون نقاط أو شرطات.
  - `1000` أصول، `2000` خصوم، `3000` حقوق ملكية، `4000` إيرادات، `5000` مصروفات/أعباء.
  - إمكانية التوسع لاحقاً إلى 5 أو 6 خانات من إعداد القالب.
- معاينة الشجرة قبل الزرع
- زر "زرع الشجرة الافتراضية" لأول إعداد
- نسخة template وإصدار version حتى يمكن ترقية القالب لاحقاً بدون كسر حسابات المشترك
- حفظ أصل كل حساب مزروع: `template_item_id`, `template_version`, وهل الحساب مخصص يدوياً
- صفحة/زر فحص صحة القالب قبل الزرع: أكواد مكررة، حسابات بلا أقسام تقارير، حسابات رقابية ناقصة، حسابات قابلة للترحيل ولها أبناء

#### شجرة الحسابات (`/accounting/accounts-tree`)
- شجرة هرمية قابلة للتوسع/الطي — يسار
- لوحة تفاصيل الحساب المختار — يمين
- فلترة بالتصنيف، القسم المالي، الحسابات الرقابية، الحسابات القابلة للترحيل، الحسابات المعطلة
- دعم أسماء عربية وفرنسية + كود رقمي `1000` style + path
- كل حساب يعرض: الكود، الاسم، التصنيف، القسم في القوائم، سطر التقرير، الرصيد المباشر، الرصيد المجمع
- **قواعد صارمة:** لا ترحيل على حساب أب، الحسابات الرقابية تحتاج sub-ledger، الحساب المقفل لا يقبل ترحيل
- إضافة حساب فرعي: يقترح الكود التالي تلقائياً، يرث التصنيف والطبيعة من الأب
- تعطيل الحساب بدلاً من حذفه إذا كانت له حركات
- عرض العلاقات: هل الحساب مستخدم في إعدادات ربط، ضريبة، بنك، صندوق، فئة منتج، أو قاعدة تلقائية
- نموذج الحساب يجب أن يحتوي تبويبات:
  - البيانات: الكود، الاسم العربي، الاسم الفرنسي، الأب، التصنيف، الطبيعة
  - التقارير: `financial_statement`, `report_section`, `statement_line_code`, `cash_flow_section`
  - الرقابة: `is_control`, `requires_subledger`, `subledger_type`, `is_reconcilable`
  - القيود: `is_postable`, `allow_manual_posting`, الحالة، سبب القفل
  - العلاقات: استعمال الحساب في إعدادات الربط والقواعد والضرائب والبنوك والمنتجات

#### فحص صحة شجرة الحسابات (`/accounting/accounts-tree/health`) — **مطلوبة**
- صفحة أو Drawer يعرض مشاكل الشجرة قبل التشغيل الفعلي
- يكشف: أكواد مخالفة، حسابات آباء قابلة للترحيل، حسابات رقابية بدون sub-ledger، حسابات مستخدمة وهي مقفلة، حسابات بلا أقسام تقارير
- يعطي زر انتقال مباشر لتعديل الحساب أو إعداد الربط المتسبب في المشكلة
- يستعمل نفس قواعد التحقق التي يستعملها الباك اند، وليس فحص واجهة فقط

#### ربط الحسابات والقواعد التلقائية (`/accounting/account-links`) — **مطلوبة**
- صفحة مركزية لكل الحسابات الافتراضية:
  - مبيعات، مردودات مبيعات، خصم ممنوح، تكلفة بضاعة، مخزون، عملاء، موردين
  - صندوق، بنك، شيكات، ضرائب محصلة، ضرائب قابلة للخصم، مصاريف شحن، فروقات جرد
- تبويب "قواعد التوليد": sale_confirmed, customer_payment, purchase_received, supplier_payment, return_received, stock_adjustment
- كل قاعدة تعرض سطورها: مدين/دائن، مصدر الحساب، مصدر المبلغ، مصدر sub-ledger، قالب الوصف
- فحص صحة القواعد قبل التفعيل
- منع حفظ ربط إلى حساب أب، حساب مقفل، حساب غير نشط، أو حساب لا يطابق التصنيف المطلوب

#### دفتر اليومية (`/accounting/journals`)
- قائمة القيود مع تبويبات: الكل | مسودة | مرحّل | معكوس
- فلتر بالسنة المالية — كل القيود تُعرض حسب السنة المختارة
- نافذة عرض القيد: رأس القيد + أسطره + إجمالي مدين/دائن + زر عكس القيد
- السنة المغلقة: القائمة تعرض بيانات فقط بدون أزرار تعديل

#### دفاتر اليومية والترقيم (`/accounting/journal-books`) — **مطلوبة**
- تعريف دفاتر: مبيعات، مشتريات، صندوق، بنك، مخزون، يدوي، افتتاح، إقفال
- لكل دفتر prefix وترقيم مستقل حسب السنة
- تحديد هل يحتاج موافقة قبل الترحيل
- منع القيود اليدوية في دفاتر النظام إذا كانت غير مسموحة

#### قيد يدوي (`/accounting/journals/new`)
- حقول الرأس: التاريخ + البيان
- اختيار دفتر اليومية + السنة/الفترة تلقائياً من التاريخ
- جدول الأسطر: Autocomplete للحساب + sub-ledger إذا مطلوب + أبعاد تحليلية + مدين/دائن + وصف
- **قواعد الترحيل:** مجموع مدين = مجموع دائن، لا يجمع مدين ودائن في نفس السطر، لا ترحيل على حساب أب، لا ترحيل على حساب رقابي بدون sub-ledger
- زر "حفظ كمسودة" + زر "ترحيل" (يُعطَّل إذا القيد غير متوازن)
- بعد الترحيل: القيد للقراءة فقط — يُعكس بقيد جديد لا بالحذف

#### الأستاذ العام (`/accounting/reports/general-ledger`) — **مطلوب**
- دفتر تفصيلي لكل الحسابات أو حساب محدد
- رصيد متراكم بعد كل حركة
- فلترة بالفترة، دفتر اليومية، المصدر، مركز التكلفة
- رابط مباشر من كل سطر إلى القيد الأصلي

#### السنوات المالية (`/accounting/fiscal-years`)
- قائمة السنوات مع حالة كل واحدة (مفتوحة/مغلقة)
- **قفل السنة بـ Stepper احترافي من 5 خطوات:**
  1. فحص الشروط: لا قيود مسودة، كل القيود متوازنة، كل الفترات مغلقة
  2. قفل الفترات الشهرية
  3. توليد قيود الإقفال (إيرادات + مصروفات → أرباح مرحلة)
  4. إنشاء أرصدة افتتاحية للسنة التالية (ميزانية فقط)
  5. تأكيد القفل مع سبب إلزامي
- السنة المغلقة: Read-only كاملة في كل واجهات المحاسبة
- إعادة الفتح: تتطلب صلاحية خاصة + سبب + تسجيل في Audit Log

#### الأرصدة الافتتاحية (`/accounting/opening-balances`)
- جدول بكل الحسابات القابلة للترحيل
- حقل مدين وحقل دائن لكل حساب
- مؤشر التوازن في الأعلى (مجموع مدين vs مجموع دائن)
- تبويب تفصيلي للحسابات الرقابية: أرصدة افتتاحية لكل زبون ومورد
- يُعطَّل إذا السنة مغلقة

#### الذمم والمطابقة (`/accounting/sub-ledgers`) — **مطلوبة**
- تبويبات: العملاء | الموردون | الضرائب | البنوك
- عرض الرصيد الرقابي مقابل مجموع sub-ledger
- كشف زبون/مورد محاسبي من القيود وليس من الطلبيات فقط
- مطابقة دفعات مع فواتير
- تنبيه إذا حساب رقابي لا يطابق تفصيله

#### الصندوق والبنك (`/accounting/cash-bank`) — **مطلوبة**
- تعريف الصناديق والحسابات البنكية وربطها بحسابات الأستاذ

- قيود تلقائية للقبض والصرف
- مطابقة بنكية لاحقاً

#### الضرائب (`/accounting/taxes`) — **مطلوبة**

- ربط كل ضريبة بحساب محصل/قابل للخصم/مستحق
- تحديد إذا السعر شامل الضريبة أو بدونها
- تقرير ضريبي شهري/سنوي

#### الأبعاد ومراكز التكلفة (`/accounting/dimensions`) — **مطلوبة**

### 9. المستخدمون (Users)
**الحالة:** ✅ واجهة منجزة — **تحديث مطلوب**

**صلاحيات جديدة:**
- نوع المستخدم: **بائع** أو **إداري**
  - البائع: يرى واجهة إضافة طلبية المبسطة (شبكة منتجات)
  - الإداري: يرى واجهة Odoo-style (سطور نصية)
  - البائع لا يؤكد الطلبيات — يرسلها للإدارة فقط

#### مبدأ إظهار الميزات والصلاحيات للجميع

**القرار:** لا تخفي الواجهة الميزات من المستخدمين. كل القوائم والأزرار والصفحات الأساسية تبقى ظاهرة حتى يتعلم المستخدم ما يستطيع النظام فعله. الفرق يكون عند الاستخدام: إذا حاول تنفيذ إجراء لا يملكه، تظهر رسالة واضحة حسب سبب المنع.

**أنواع المنع المطلوبة:**

1. **الميزة غير موجودة في خطة الاشتراك**
   - تظهر الميزة والزر بشكل طبيعي.
   - عند الضغط يظهر Modal ترقية الخطة.
   - الرسالة تكون مثل: "هذه الميزة متاحة في الخطة الاحترافية. راسل مالك المنصة أو قم بترقية الخطة."
   - أمثلة: الذكاء الاصطناعي، المحاسبة المتقدمة، تكاملات مخصصة، حدود مستخدمين أعلى.

2. **المستخدم داخل شركة المشترك لا يملك الصلاحية**
   - تظهر الميزة، لكن عند الضغط يظهر تنبيه: "ليس لديك صلاحية لتنفيذ هذا الإجراء."
   - الرسالة توجهه إلى مالك الشركة/الإدارة/المسؤول حسب الإعداد: "تواصل مع مالك الشركة أو الإدارة لمنحك الصلاحية."
   - أمثلة: تأكيد طلبية، تعديل أسعار، إدارة المستخدمين، ترحيل قيد، إغلاق سنة مالية.

3. **الصلاحية تعتمد على شريك مرتبط**
   - تظهر ميزة الشراكة، لكن عند استخدام جزء لم يمنحه الشريك تظهر رسالة: "الشريك لم يمنحك هذه الصلاحية."
   - الرسالة توجهه إلى الشريك: "تواصل مع الشريك ليمنحك صلاحية عرض الأسعار/إنشاء طلب شراء/نسخ الأصناف."
   - أمثلة: عرض أسعار الشريك، نسخ الأصناف، إنشاء شراء مرتبط، تحويل بيع إلى شراء مقابل.

4. **البيانات خارج نطاق المستخدم**
   - تظهر الصفحة، لكن عند فتح سجل خارج نطاقه تظهر رسالة نطاق: "لا يمكنك الوصول لهذا السجل لأنه ليس ضمن نطاقك."
   - أمثلة: بائع يحاول فتح طلبية بائع آخر إذا سياسة الشركة تمنع ذلك.

**قواعد الواجهة:**
- لا يستخدم النظام إخفاء القوائم كوسيلة أساسية للصلاحيات.
- الأزرار المهمة تبقى مرئية، ويمكن أن تحمل علامة قفل صغيرة أو badge "يتطلب ترقية" بدون حذفها.
- الضغط على إجراء ممنوع يفتح `AccessDeniedDialog` أو `PlanUpgradeDialog` حسب السبب.
- الرسالة يجب أن تحتوي سبب المنع، الجهة التي يمكن التواصل معها، وزر إجراء مناسب: "طلب صلاحية"، "فتح الدعم"، "ترقية الخطة"، "التواصل مع الشريك".
- الواجهة لا تعتمد على نفسها للأمان؛ الباك إند يعيد نفس قرار المنع حتى لو حاول المستخدم استدعاء API مباشرة.

**الفلسفة:** الإشعارات لا تُنشأ مباشرة من الواجهات، بل من Domain Events. كل عملية مهمة تكتب event، ثم Notification Engine يقرر هل يولد إشعارًا، لمن، وبأي قناة، وهل يتم تجميعه أو منعه بالتكرار.

#### تقسيم الإشعارات

1. **إشعارات المنصة (`platform`)**
   - تظهر لمالك المنصة وفريق الدعم/الفوترة.
   - أمثلة:
     - قرب انتهاء اشتراك مشترك خلال 30/15/7/1 يوم.
     - اشتراك انتهى أو أصبح past_due.
     - فشل إنشاء schema لمشترك.
     - مشترك تجاوز حد المستخدمين/الطلبيات/التخزين في خطته.
     - طلب ربط جديد بين مشتركين يحتاج متابعة.
     - مشترك موقوف حاول الدخول.

2. **إشعارات داخل المشترك (`tenant`)**
   - تظهر لمستخدمي شركة واحدة فقط.
   - أمثلة:
     - إضافة صنف جديد.
     - تغيير سعر صنف.
     - انخفاض المخزون تحت حد التنبيه.
     - نفاد المخزون.
     - طلبية جديدة من بائع.
     - تأكيد/رفض طلبية.
     - إنشاء مرتجع.
     - استلام دفعة زبون.
     - إنشاء قيد محاسبي تلقائي أو فشل توليده.
     - تعديل صلاحيات مستخدم.
     - محاولة دخول فاشلة أو تعطيل مستخدم.

3. **إشعارات بين مشتركين (`inter_tenant`)**
   - تمر عبر `platform` ولا تفتح وصول مباشر بين schemas.
   - أمثلة:
     - مشترك أرسل طلب ربط باستخدام كود.
     - مشترك قبل أو رفض طلب الربط.
     - صلاحيات الربط تغيرت.
     - شريك أوقف مشاركة المخزون.

- كل إشعار له `severity`: info / success / warning / critical / action_required.

- **الخصوصية:** إشعارات بين مشتركين لا تكشف بيانات غير مصرح بها من شركة أخرى.
- **الاحتفاظ والتنظيف:** سياسات حفظ مختلفة للإشعارات العادية، الأمان، outbox، والمؤرشفة.

### 12. نظام الدعم والشات بين المالك والمشتركين
**الحالة:** ✅ خطة + صفحات Mock + Schema، 🔄 ربط Backend لاحقًا

**الفلسفة:** الدعم ليس رسالة بريد منفصلة، بل تذكرة لها محادثة كاملة بين المشترك ومالك المنصة أو فريق الدعم. كل رسالة ومرفق وتغيير حالة محفوظ وقابل للإشعار والتدقيق.

#### واجهات مطلوبة/مجهزة

- `src/layouts/support/index.js` — صفحة دعم داخل تطبيق المشترك: قائمة التذاكر، فتح تذكرة، شات، إرسال صورة، إرسال تسجيل صوتي.
- `src/layouts/owner/support/index.js` — مركز دعم في لوحة المالك: طابور التذاكر، فلترة حسب المشترك/الأولوية، شات مع المشترك.
- رابط المشترك: `/support`
- رابط المالك: `/owner/support`

#### سلوك التذكرة

- المشترك يفتح تذكرة من صفحة الدعم ويحدد: العنوان، التصنيف، الأولوية، الوصف، ومرفقات اختيارية.
- بعد فتح التذكرة تنشأ محادثة مرتبطة بها بين المشترك وفريق الدعم/المالك.
- يمكن للطرفين إرسال رسالة نصية، صورة/لقطة شاشة، تسجيل صوتي، وملف لاحقًا.
- حالات التذكرة: `open`, `waiting_owner`, `waiting_tenant`, `resolved`, `closed`.
- المستخدمون لا يغيرون `waiting_owner` أو `waiting_tenant` يدوياً.
- المسموح يدوياً فقط: فتح تذكرة، إغلاق تذكرة، إعادة فتح تذكرة.
- عند إرسال رسالة من المشترك تتحول الحالة تلقائياً إلى `waiting_owner`.
- عند إرسال رسالة من المالك/الدعم تتحول الحالة تلقائياً إلى `waiting_tenant`.
- عند الإغلاق تصبح `closed`، ولا يمكن إرسال رسائل جديدة حتى يعاد فتحها.
- الأولويات: `low`, `normal`, `high`, `urgent`.
- كل رسالة تحفظ حالة القراءة لكل طرف حتى تظهر badge في صفحات الدعم والإشعارات.
- إغلاق التذكرة لا يحذف المحادثة أو المرفقات؛ تبقى ضمن سجل الدعم وسياسة الاحتفاظ.

#### الإشعارات

- عند فتح تذكرة أو إرسال رسالة من المشترك، يصل إشعار للمالك أو فريق الدعم.
- عند رد المالك أو الدعم، يصل إشعار للمشترك صاحب التذكرة.
- عند إغلاق التذكرة أو إعادة فتحها، يصل إشعار للطرف الآخر.
- الإشعارات تستخدم `action_url` لفتح التذكرة مباشرة.

### 13. قفل التحرير ومنع التعديل المتزامن
**الحالة:** ✅ خطة + Schema + مكوّن واجهة Mock، 🔄 ربط Backend لاحقًا

**المرجع التفصيلي:** [RESOURCE_LOCKING_BLUEPRINT.md](/Users/mw/Downloads/allal-article/RESOURCE_LOCKING_BLUEPRINT.md:1)

**الفلسفة:** عندما يدخل مستخدم لتعديل سجل حساس، يأخذ Lock قصير العمر. إذا حاول مستخدم آخر تعديل نفس السجل، يرى أن فلان يعمل عليه الآن ويدخل وضع عرض فقط. هذا يمنع تضارب التعديلات في الطلبيات، المشتريات، المخزون، والمحاسبة.

#### السلوك المطلوب

- عند فتح شاشة تعديل، يستدعي الفرونت `acquire lock`.
- إذا القفل متاح، يسمح النظام بالتعديل ويبدأ `heartbeat` كل 30 ثانية.
- إذا القفل مشغول، تعرض الواجهة اسم المستخدم، وقت بداية التحرير، وآخر نشاط.
- المستخدم الثاني يستطيع القراءة فقط، وتتعطل أزرار الحفظ/التأكيد/الترحيل حسب السياسة.
- عند الحفظ أو الإلغاء أو مغادرة الصفحة، يتم `release lock`.
- إذا انقطع الاتصال، ينتهي القفل تلقائيًا بعد مدة قصيرة.
- المدير يستطيع `take over` فقط إذا كانت السياسة تسمح وبصلاحية واضحة.

#### أين يطبق

- **طلبيات البيع:** تعديل الطلبية، مراجعة الإدارة، التأكيد/الرفض، تحويل إلى فاتورة طريق، إنشاء مرتجع.
- **أوامر الشراء:** تعديل الأمر، التأكيد، الاستلام، ربط فاتورة المورد، تسجيل دفعة مرتبطة.
- **المخزون:** الجرد، التسوية، التحويل بين مخازن، تعديل حد التنبيه.
- **الأصناف والأسعار:** تعديل الصنف، تغيير السعر/التكلفة، ربط حسابات الصنف.
- **المحاسبة:** القيد اليدوي، الترحيل، العكس، فتح الأرصدة، إقفال فترة أو سنة، إعدادات الحسابات.
- **المستخدمون:** تعديل الصلاحيات، الأدوار، تعطيل المستخدم.
- **الشركاء:** قبول/رفض الربط، تغيير صلاحيات الربط، إلغاء الربط.
- **المنصة:** تعديل خطة، إيقاف/تجديد اشتراك، إعادة provisioning.

#### قواعد Backend

- القفل تطبيقي وليس transaction مفتوحة في قاعدة البيانات.
- كل endpoint حساس يتحقق من `lock_token` إذا كانت سياسة المورد تتطلب lock.
- القفل لا يغني عن `version` أو `updated_at` عند الحفظ؛ يجب منع overwrite إذا تغير السجل.
- كل acquire/release/expire/takeover يسجل في `resource_lock_events` و audit log.
- التولي القسري يولد إشعارًا للمستخدم الأول.

#### الواجهة المطلوبة/المجهزة

- `src/components/ResourceLockBanner/index.js` — يعرض من يعمل على السجل ويعطل التعديل.
- `src/data/mock/resourceLocksMock.js` — أمثلة أقفال تحرير.
- لاحقًا `useResourceLock` لإدارة acquire/heartbeat/release.
- لاحقًا `LockedActionButton` لتعطيل الأزرار مع سبب واضح.

### قرار الباك إند طويل المدى

تم اعتماد Spring Boot كخيار معماري مستقبلي لأن النظام ليس CRUD بسيطًا، بل منصة تشغيل داخلية فيها طلبيات، مخزون، مدفوعات، فواتير طريق، صلاحيات، وسجلات تدقيق. الأولوية هنا هي القوة، الأمان، قابلية الصيانة، ودقة المعاملات على المدى الطويل، وليس سرعة بناء MVP.

النمط المعتمد هو `Modular Monolith`: مشروع باك إند واحد منظم داخليًا إلى وحدات مثل `orders`, `inventory`, `customers`, `payments`, `roadinvoices`, `notifications`, `audit`, `settings`. لا يتم البدء بـ microservices حتى لا يزيد التعقيد قبل الحاجة الفعلية.

PostgreSQL هو الخيار الرسمي للبيانات بسبب قوة العلاقات، القيود، الفهارس، و transactions. كل تعديل في قاعدة البيانات يجب أن يتم عبر Flyway migration واضح ومؤرشف.

### قرار فصل بيانات المشتركين

النظام يُعامل كمنصة SaaS فيها مالك منصة ومشتركون مستقلون. لذلك لا تُخزَّن بيانات كل الشركات في نفس الجداول التشغيلية العامة بدون عزل.

القرار المعتمد:
- `platform` schema واحد لإدارة المنصة: المشتركين، الخطط، الاشتراكات، أحداث التهيئة، مستخدمي مالك المنصة، وإعدادات المنصة العامة.
- schema منفصل لكل مشترك باسم آمن مثل `tenant_a1b2c3d4`.
- كل بيانات التشغيل الخاصة بالمشترك تكون داخل schema الخاص به: المستخدمون الداخليون، الزبائن، الأصناف، الطلبيات، المخزون، المحاسبة، التقارير، الإعدادات، وسجلات التدقيق.
- لا تقبل واجهات الـ API اسم schema أو tenant من جسم الطلب. تحديد المشترك يتم من JWT/subdomain/header موثوق، ثم يضبط الباك إند سياق المشترك داخليًا.
- شبكة الشركاء بين المشتركين تُدار من `platform` عبر صلاحيات صريحة، ولا تعطي وصولًا مباشرًا بين schemas بدون Service Layer تتحقق من الموافقة والصلاحيات.

تسلسل إنشاء مشترك جديد:
1. إنشاء سجل في `platform.tenants`.
2. توليد `schema_name` آمن وفريد.
3. إنشاء schema PostgreSQL للمشترك.
4. تطبيق migrations الخاصة بالمشترك على ذلك schema.
5. زرع البيانات الأساسية: الولايات، الأدوار، الصلاحيات، المخزن الافتراضي، شجرة الحسابات.
6. إنشاء أول مستخدم admin داخل schema المشترك.
7. تسجيل حدث في `platform.tenant_provisioning_events`.
